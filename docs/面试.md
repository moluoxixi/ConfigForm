# 前端架构面试故事（医疗行业，5 年经验版）

> 使用方式：每个场景都按「业务目标-踩坑-定位-决策-结果」来讲。内容是合理虚构，但技术链路和指标是自洽的，适合大厂前端/前端架构岗位。

## 场景 1：门诊病历首页高并发下的首屏与运行时性能瓶颈

### 1) 业务背景与目标
- 背景：从 0 到 1 上线门诊病历系统，医生在一个页面同时处理主诉、处方、检验报告、既往史，字段 320+，检查报告列表单日可达 2 万条。
- 目标：工作日早高峰（8:30-10:00）保持可用，首屏可交互时间 `TTI < 3s`，输入卡顿 `INP < 200ms`，页面可用率 `>= 99.9%`。

### 2) 初始方案
- 技术选型：`Vue 3 + TypeScript + Vite + Pinia + Axios`，表单渲染用 Schema 驱动，考虑后续多院区快速复制。
- 当时判断：先保证业务功能闭环，首版采用「一次性拉全量病历+报告数据」，页面通过多个 `watch` 做联动。

### 3) 踩坑表现
- 线上症状：首屏白屏 5-8 秒，医生输入体温时光标抖动，切换 tabs 会明显掉帧。
- 监控数据：`TTI P95 = 6.7s`，输入延迟 `INP P95 = 480ms`，页面内存峰值接近 900MB（长时间问诊会话）。
- 用户影响：早高峰诊室排队变长，医生反馈“录入一行字要等半拍”。

### 4) 排查过程
- 前端性能画像：`Web Vitals + Performance 面板 + Vue DevTools`，定位主线程长任务和重复渲染。
- 关键发现：
  - 首屏阶段同步执行了 3 次大对象 `deep clone`。
  - `watch` 粒度过粗，单次输入触发 1200+ 依赖计算。
  - 报告列表直接渲染 4000+ DOM 节点，没有虚拟化。
- 联动链路排查：结合 Sentry breadcrumb 和接口耗时，确认并非后端慢，而是前端渲染/计算阻塞。

### 5) 根因分析
- 技术根因：
  - 全量拉取 + 全量渲染导致首屏工作量过大。
  - 状态联动设计偏“全局响应”，缺少局部更新边界。
  - 复杂对象频繁序列化和深拷贝。
- 流程根因：
  - 首版没有设定明确性能预算。
  - 评审更偏功能正确性，缺少性能走查清单。

### 6) 解决方案
- 短期止血（1 周内）：
  - 报告区改成分段加载 + 虚拟滚动。
  - 联动改为字段级订阅，去掉大范围 `deep watch`。
  - 增加骨架屏和关键路径优先加载，非核心模块延迟挂载。
- 长期治理（2 个迭代）：
  - 建立“首屏关键路径”分层：基础病历、处方、报告按优先级异步装配。
  - 引入性能预算：`bundle、TTI、INP` 都设阈值并接入 CI 门禁。
  - 高频计算迁移到 Web Worker（病历差异比对、风险规则预计算）。

### 7) 架构取舍
- 备选方案 A：继续 CSR，只做局部优化。
- 备选方案 B：SSR 首屏 + 客户端激活。
- 备选方案 C：CSR + 分层数据加载 + 组件级虚拟化（最终选择）。
- 决策理由：
  - SSR 在医院内网和多终端环境下改造成本高，联调周期长。
  - C 方案投入最小且对现有 Vue 3 架构改动可控，能快速回收性能收益。

### 8) 结果量化
- `TTI P95` 从 `6.7s` 降到 `2.4s`。
- `INP P95` 从 `480ms` 降到 `130ms`。
- 页面内存峰值从 `900MB` 降到 `420MB`。
- 医生端“录入卡顿”工单 4 周内下降 `76%`。

### 9) 复盘沉淀
- 建立《医疗表单性能基线》：字段规模、联动复杂度、列表渲染上限。
- 落地性能 SOP：需求评审必须写“性能风险点 + 预算”。
- 平台化能力：沉淀虚拟列表、字段级订阅、长任务告警 SDK。

### 10) 面试表达版本
- 30 秒电梯版：
  - “我负责从 0 到 1 的门诊病历前端，首版因为全量拉取和粗粒度联动导致 TTI 到 6.7 秒、输入延迟接近 500ms。我们先做分层加载和虚拟滚动止血，再把联动重构成字段级订阅，并在 CI 加性能门禁。最后 TTI 降到 2.4 秒，卡顿工单下降 76%。”
- 3 分钟完整版（STAR）：
  - `S`：门诊病历首发，页面字段多、报告量大，早高峰并发下医生录入明显卡顿。
  - `T`：在不延期上线的前提下，把首屏和输入体验拉回可用区间。
  - `A`：我主导性能排查，先拆分渲染瓶颈（DOM 过多）和联动瓶颈（watch 粒度过粗），第一周完成分段加载+虚拟滚动+骨架屏，第二阶段做字段级订阅和 Worker 迁移，并推动性能预算接入 CI。
  - `R`：TTI 从 6.7 秒到 2.4 秒，输入延迟降到 130ms，投诉显著下降，后续同类页面复用同一套性能治理模板。
- 面试官可能追问 3 个问题 + 最佳回答：
  - 问题 1：为什么不用 SSR 一步到位？
  - 回答：当时医院终端环境复杂，SSR 需要改网关、缓存和同构容错，短期收益不如分层 CSR。我们先用低风险方案把核心指标拉下来，再评估局部 SSR。
  - 问题 2：字段级订阅怎么避免维护成本变高？
  - 回答：定义统一依赖声明协议，联动规则写成可测试的纯函数，并配了可视化依赖图。这样复杂度从“隐式 watch”变成“显式依赖”。
  - 问题 3：性能门禁如何避免阻塞业务迭代？
  - 回答：门禁分级，主干只卡 P95 严重退化，其他指标先告警不阻塞。稳定两周后再提高阈值，保证节奏和质量平衡。

### 可背诵话术
“这个项目我最大的成长是：以前我只盯‘功能跑通’，后来我把性能当成架构问题。先做低风险止血，再做系统治理，把一次事故变成团队标准。”

## 场景 2：病历与处方联调中的数据一致性和状态管理失控

### 1) 业务背景与目标
- 背景：从 0 到 1 搭建病历、处方、检验三模块联动页面，不同子系统由不同团队交付。
- 目标：保证“医生看到的数据”和“最终提交的数据”一致，杜绝错单、漏单；提交成功率 `>= 99.95%`。

### 2) 初始方案
- 方案：新增模块用 `Pinia`，历史模块保留 `Vuex`，组件内部保留部分本地状态。
- 当时考虑：为了不影响上线节奏，采用渐进迁移，先兼容旧逻辑。

### 3) 踩坑表现
- 线上症状：医生修改过敏史后，处方页偶现未同步；重复点击提交会产生两张相同处方。
- 监控指标：提交失败率从 `0.2%` 升到 `1.8%`，重复订单告警一周 37 起。
- 用户影响：药房退单率上升，临床流程被迫人工二次核对。

### 4) 排查过程
- 工具链：`Pinia action 日志 + Vuex mutation 日志 + 接口 traceId` 串联。
- 关键发现：
  - 同一字段有 3 个“真值来源”：Pinia、Vuex、本地 `ref`。
  - 跨模块事件通过 `window bus` 广播，存在丢事件和乱序。
  - 提交缺少幂等键，网络抖动导致重试时重复落单。

### 5) 根因分析
- 技术根因：
  - 状态单一数据源原则被破坏。
  - 跨团队字段协议不统一（同一语义字段命名不同）。
  - 提交流程缺少“版本号 + 幂等控制”。
- 流程根因：
  - 联调阶段只测“正常路径”，没有覆盖并发点击和弱网重试。
  - 接口契约评审不严格，字段变更缺少通知机制。

### 6) 解决方案
- 短期止血：
  - 提交按钮加前端防抖与进行中锁。
  - 服务端加 `idempotencyKey`，重复请求直接返回首单结果。
  - 紧急修复关键字段同步逻辑，统一走 Pinia。
- 长期治理：
  - 建立“病历域模型 Store”，Pinia 成为唯一写入口，Vuex 仅做兼容只读。
  - 引入版本戳（`recordVersion`）+ 提交前后比对，防止旧数据覆盖新数据。
  - 落地 BFF 字段映射层，统一字段协议与校验。

### 7) 架构取舍
- 备选方案 A：全面回退 Vuex，推迟 Pinia 迁移。
- 备选方案 B：Pinia 与 Vuex 长期并存，靠规范约束。
- 备选方案 C：Pinia 单一真值源 + BFF 协议收口（最终选择）。
- 决策理由：
  - A 方案短期稳但技术债更重。
  - B 方案治理成本高，长期必然再出一致性问题。
  - C 方案一次性明确边界，可测试性最好。

### 8) 结果量化
- 提交成功率从 `98.2%` 提升到 `99.97%`。
- 重复处方告警从每周 `37` 起降到 `0-1` 起。
- 联调缺陷平均修复时间从 `2.5 天` 降到 `0.8 天`。

### 9) 复盘沉淀
- 输出《医疗主数据一致性规范》：单一真值源、字段协议、版本控制、幂等策略。
- 建立联调清单：弱网、重复提交、跨页切换、回填覆盖等场景必须跑。
- 接入“状态变更审计日志”，线上追溯时间从小时级降到分钟级。

### 10) 面试表达版本
- 30 秒电梯版：
  - “我在病历处方联调里踩过一个典型坑：Pinia、Vuex、本地状态并存导致数据源混乱，出现漏同步和重复提交。我们短期用提交锁和幂等键止血，长期把 Pinia 收敛成唯一真值源，并加版本戳和字段协议层，提交成功率做到 99.97%。”
- 3 分钟完整版（STAR）：
  - `S`：多团队并行开发病历和处方页面，状态管理混用，线上出现一致性问题。
  - `T`：把错单风险降到可控，同时不影响当月上线节点。
  - `A`：我先串联前端 action 日志和后端 trace，定位到多真值源和无幂等控制；第一阶段上线提交锁+幂等键，第二阶段推进 Pinia 单一写入和 BFF 协议映射，最后用版本戳防覆盖。
  - `R`：重复订单基本清零，提交成功率接近 100%，联调效率明显提升。
- 面试官可能追问 3 个问题 + 最佳回答：
  - 问题 1：Pinia 比 Vuex 的关键收益是什么？
  - 回答：类型推断和模块组织更轻量，迁移后 action 链路更清晰；但核心收益不是“库本身”，而是借迁移机会重建单一真值源。
  - 问题 2：怎么防止跨团队字段协议再次失控？
  - 回答：字段字典放到 BFF 层统一管理，前端 CI 增加契约测试，字段变更必须走版本化评审。
  - 问题 3：为什么要前后端都做幂等？
  - 回答：前端防重只解决“误触发”，后端幂等才能兜住重试、超时、网络抖动等系统性重复请求。

### 可背诵话术
“这个坑让我意识到：状态管理不是选 Pinia 还是 Vuex，而是先定义谁能写、谁能读、何时提交。架构边界不清，工具再新也会出错。”

## 场景 3：线上白屏与崩溃的稳定性治理（含终端兼容）

### 1) 业务背景与目标
- 背景：医院管理系统首批接入 40+ 科室终端，包含新旧混合浏览器和自助机。
- 目标：白屏率 `< 0.1%`，JS 崩溃率 `< 0.3%`，核心问诊链路可用率 `>= 99.95%`。

### 2) 初始方案
- 方案：微前端拆分（主应用 + 病历子应用 + 处方子应用），统一通过动态加载子包。
- 当时考虑：团队并行开发效率高，业务边界清晰。

### 3) 踩坑表现
- 线上症状：发布后部分终端打开即白屏，控制台报 `Unexpected token '?.'`；部分设备切路由后整页崩溃。
- 监控数据：白屏率峰值 `1.9%`，错误集中在旧版 Chromium 内核终端。
- 用户影响：自助机无法开单，前台转人工，窗口拥堵。

### 4) 排查过程
- 监控定位：Sentry 按 `UA + 版本 + 子应用` 聚合，发现问题集中在“旧内核 + 某子应用新构建产物”。
- 发布回溯：比对本次构建配置，发现目标浏览器降级配置在子应用被误删。
- 复现场景：本地用旧内核容器复现，确认可选链和部分 polyfill 缺失导致解析失败。

### 5) 根因分析
- 技术根因：
  - 子应用构建目标与主应用不一致，产物语法超出终端能力。
  - 缺少“子应用加载失败兜底页”和 error boundary。
  - 路由切换阶段异常未被隔离，导致整页崩溃。
- 流程根因：
  - 发布前兼容性验证只覆盖主应用，漏测子应用独立构建产物。
  - 灰度分层不细，问题版本扩散过快。

### 6) 解决方案
- 短期止血：
  - 10 分钟内回滚子应用版本。
  - 主应用加子应用加载失败兜底页，保证核心链路可退化使用。
  - 补发兼容构建（降级语法 + 补齐 polyfill）。
- 长期治理：
  - 建立浏览器兼容矩阵（科室终端型号、内核版本、能力清单）。
  - 主/子应用统一 `browserslist` 与构建校验脚本，发布前自动比对。
  - 引入前端 SLO 和稳定性例会，跟踪 MTTR、白屏率、崩溃率。

### 7) 架构取舍
- 备选方案 A：放弃微前端，回归单体应用。
- 备选方案 B：保留微前端，强化治理和容错（最终选择）。
- 备选方案 C：双轨，核心链路单体、非核心微前端。
- 决策理由：
  - A 方案会明显降低并行研发效率。
  - B 方案在治理到位后可兼顾效率和稳定性。
  - C 方案复杂度更高，当前阶段收益不如 B。

### 8) 结果量化
- 白屏率从 `1.9%` 降到 `0.07%`。
- JS 崩溃率从 `0.8%` 降到 `0.21%`。
- 故障恢复时间（MTTR）从 `46 分钟` 降到 `11 分钟`。

### 9) 复盘沉淀
- 固化《线上故障 15 分钟处置 SOP》：告警分级、回滚权限、沟通模板。
- 增加“终端兼容回归流水线”，每次发布前自动跑关键页面快照和冒烟。
- 建立子应用健康看板：加载成功率、初始化耗时、错误率实时可见。

### 10) 面试表达版本
- 30 秒电梯版：
  - “我经历过一次典型白屏事故：微前端子应用构建目标配置漂移，旧终端解析失败。我们先回滚止血，再补加载兜底和兼容构建，后续做了主子应用构建一致性校验。白屏率从 1.9% 降到 0.07%，MTTR 降到 11 分钟。”
- 3 分钟完整版（STAR）：
  - `S`：医院终端环境复杂，发布后出现白屏和崩溃，影响挂号开单效率。
  - `T`：快速恢复可用，并建立可持续稳定性机制。
  - `A`：我先基于 Sentry 聚合定位问题终端和子应用，10 分钟内回滚；随后补齐兼容构建和兜底页，最后推动兼容矩阵、构建一致性脚本和灰度策略升级。
  - `R`：核心稳定性指标回到目标线内，后续 3 个月未再出现同类大面积故障。
- 面试官可能追问 3 个问题 + 最佳回答：
  - 问题 1：白屏率怎么统计才可信？
  - 回答：采用“首屏渲染标记 + 心跳兜底 + 采样校验”三段式，过滤机器人和短会话，保证统计口径稳定。
  - 问题 2：error boundary 在 Vue 里怎么落地？
  - 回答：组件级用 `errorCaptured`，应用级用全局错误处理与路由守卫联动，子应用加载失败直接切换到降级视图。
  - 问题 3：兼容矩阵维护成本高，如何持续？
  - 回答：只维护“真实在网终端”，每月由运维同步资产清单，流水线按清单自动选择回归集。

### 可背诵话术
“我对稳定性的理解是：不是不出事故，而是事故发生后能快速止血、准确复盘、系统性防复发。”

## 场景 4：医疗数据安全、权限分级与合规整改

### 1) 业务背景与目标
- 背景：互联网问诊 + 电子处方上线，涉及身份证号、手机号、诊断、处方等敏感信息。
- 目标：满足医院数据安全审计要求，做到最小权限、全链路审计、敏感字段受控展示。

### 2) 初始方案
- 方案：前端按角色控制菜单和按钮，接口 token 存在 `localStorage`，日志默认打印请求参数。
- 当时考虑：实现快，能先让业务跑起来。

### 3) 踩坑表现
- 问题暴露：安全测试发现“直接输入 URL 可访问越权页面”；前端日志含完整身份证号；客服录屏中存在敏感信息明文展示。
- 影响：合规评审被打回，项目上线窗口延期两周。

### 4) 排查过程
- 安全联查：前端、后端、安全团队做联合走查。
- 关键发现：
  - 前端仅做展示层权限，后端缺少字段级权限校验。
  - token 长期有效且可被 XSS 场景读取。
  - 诊断详情页面缺少水印与导出审计。

### 5) 根因分析
- 技术根因：
  - 权限模型停留在 RBAC 菜单级，未覆盖字段级和数据级。
  - 安全基线（日志脱敏、会话策略、导出审计）未内建到模板。
- 流程根因：
  - 安全评审介入太晚，接近提测才发现系统性问题。
  - 需求文档没有明确“哪些字段属于敏感级别”。

### 6) 解决方案
- 短期止血：
  - 立即下线敏感日志打印，前端统一脱敏函数（身份证、手机号、姓名）。
  - 关闭高风险入口，越权页面强制重定向。
  - token 改为短时效 + HttpOnly Cookie。
- 长期治理：
  - 建立 `RBAC + ABAC` 混合模型：角色控制功能，属性控制字段可见与可编辑。
  - 接口层强制做字段级鉴权，前端只做“展示增强”，不做最终授权判断。
  - 全链路审计：查看、导出、打印都记录操作人、时间、对象、原因。
  - 敏感信息策略平台化：水印、脱敏、复制限制、截图提醒可配置。

### 7) 架构取舍
- 备选方案 A：纯前端权限控制，后端只返回全量数据。
- 备选方案 B：后端强鉴权 + 前端做最小展示（最终选择）。
- 备选方案 C：网关统一脱敏，前端无感接入。
- 决策理由：
  - A 方案不满足医疗合规。
  - C 方案适合通用场景，但业务字段规则复杂，仍需应用层协同。
  - B 方案职责最清晰，安全边界明确。

### 8) 结果量化
- 越权缺陷从 `高危 6 项` 降为 `0`。
- 敏感日志命中率从 `3.2%` 降为 `0`。
- 合规评审一次通过，项目按新排期上线。

### 9) 复盘沉淀
- 输出《医疗前端安全基线》：权限、会话、脱敏、审计、导出。
- 把安全检查接入 CI：静态扫描 + 关键路由权限回归测试。
- 需求模板新增“敏感字段分级表”，评审时必须确认。

### 10) 面试表达版本
- 30 秒电梯版：
  - “我们在问诊处方项目里被安全评审打回过，核心问题是前端权限做得太重、后端鉴权不足。我推动改成后端强鉴权+前端最小展示，并建立字段级 ABAC 和全链路审计，最终高危越权清零，合规一次通过。”
- 3 分钟完整版（STAR）：
  - `S`：上线前安全测试发现越权访问和敏感数据暴露风险。
  - `T`：在不推翻业务功能的前提下完成合规整改。
  - `A`：先下线敏感日志和高风险入口止血，再重构权限模型，明确前后端职责边界；同时推动脱敏、水印、审计能力平台化，纳入开发模板和 CI。
  - `R`：高危漏洞清零，审计通过，后续新模块按基线落地，整改成本明显下降。
- 面试官可能追问 3 个问题 + 最佳回答：
  - 问题 1：为什么要 ABAC，不够复杂吗？
  - 回答：医疗场景同角色常有“科室、病区、患者归属”差异，RBAC 只能控功能，ABAC 才能精细到数据与字段。
  - 问题 2：前端还能做哪些安全价值？
  - 回答：前端做风险前置和用户态约束，比如脱敏展示、操作二次确认、可疑行为告警，但最终授权一定在后端。
  - 问题 3：如何平衡安全与效率？
  - 回答：把安全能力平台化，变成默认模板和自动化检查，避免每个项目重复造轮子。

### 可背诵话术
“安全这件事我踩过坑后最大的变化是：不再把它当上线前的检查项，而是当架构设计的第一天就要定义的边界。”

## 场景 5：构建部署、包体积与成本优化（Vite/Webpack 双栈）

### 1) 业务背景与目标
- 背景：医院管理系统持续扩展，微前端子应用增多，老模块在 Webpack，新模块用 Vite，构建和发布成本快速上升。
- 目标：提升发布效率、降低首包体积和带宽成本，确保院内弱网也能稳定加载。

### 2) 初始方案
- 方案：保持双构建链路并行，先功能优先；依赖由各子应用独立管理。
- 当时考虑：迁移风险小，团队改造阻力小。

### 3) 踩坑表现
- 问题表现：
  - 首包体积超过 `14MB`（含重复依赖），首次加载慢。
  - CI 全量构建平均 `26 分钟`，高峰时排队更久。
  - 月度 CDN/专线带宽成本持续上涨。

### 4) 排查过程
- 工具：`webpack-bundle-analyzer`、`rollup visualizer`、CI 构建日志。
- 关键发现：
  - UI 库和日期库多版本并存，重复打包严重。
  - 历史图片资源未压缩，部分 SVG 被当位图处理。
  - 缓存策略不合理，静态资源变更导致大范围缓存失效。

### 5) 根因分析
- 技术根因：
  - 依赖治理缺失，`shared` 策略不统一。
  - 构建产物缺少分层缓存设计（runtime/vendor/business）。
  - 流水线没有增量构建与并行策略。
- 流程根因：
  - 团队只关注“能发版”，未持续追踪构建和成本指标。
  - 缺少统一“包体积预算”。

### 6) 解决方案
- 短期止血：
  - 热点模块优先做按需加载和路由级懒加载。
  - 统一图片压缩链路，SVG 改 sprite 方案。
  - 关键依赖锁版本并去重。
- 长期治理：
  - Webpack 子应用开启 `Module Federation shared`，公共库强制单例。
  - Vite 子应用按业务边界 `manualChunks`，拆分稳定 vendor。
  - 引入 `pnpm workspace + changesets` 管理依赖升级节奏。
  - CI 做增量构建和并行任务，构建缓存持久化。

### 7) 架构取舍
- 备选方案 A：全量迁移到 Vite，一次性改完。
- 备选方案 B：继续双栈但建立统一依赖与产物治理（最终选择）。
- 备选方案 C：维持现状，只做局部懒加载。
- 决策理由：
  - A 方案收益高但窗口期太长，风险不可控。
  - C 方案见效慢且无法解决根因。
  - B 方案可分阶段落地，收益和风险平衡最好。

### 8) 结果量化
- 首包体积从 `14MB` 降到 `7.2MB`（降幅约 `49%`）。
- CI 平均构建时间从 `26 分钟` 降到 `11 分钟`。
- 月度带宽成本下降 `31%`。
- 首屏加载时间在院内弱网下缩短 `38%`。

### 9) 复盘沉淀
- 输出《前端构建治理手册》：依赖收口、分包策略、缓存策略、预算阈值。
- 建立成本看板：每周跟踪包体积、构建时长、带宽费用。
- 新项目模板默认开启懒加载、资源压缩、体积告警。

### 10) 面试表达版本
- 30 秒电梯版：
  - “我们在双构建链路下遇到包体积和构建时间失控的问题。我主导做依赖收口、shared 单例、Vite/webpack 分包治理和 CI 增量构建，首包从 14MB 降到 7.2MB，构建从 26 分钟降到 11 分钟，带宽成本下降 31%。”
- 3 分钟完整版（STAR）：
  - `S`：业务扩张后前端成本上升，发布慢、加载慢、费用高。
  - `T`：不影响交付节奏的前提下，把构建和体积控制在可持续区间。
  - `A`：我先做数据化诊断，定位重复依赖和分包问题；短期先压热点路径，长期推进 shared 策略、manualChunks、CI 增量并行，并把预算接入门禁。
  - `R`：性能和成本指标都显著改善，团队形成长期治理机制。
- 面试官可能追问 3 个问题 + 最佳回答：
  - 问题 1：为什么不一次性全迁 Vite？
  - 回答：医疗系统发布窗口受限，一次性迁移风险过高。我们采用收益优先的分阶段策略，先把最大成本点打掉。
  - 问题 2：shared 单例会不会引入版本冲突？
  - 回答：会，所以我们定义“可单例白名单”，并配兼容测试；不稳定依赖不强制单例。
  - 问题 3：如何让体积治理可持续？
  - 回答：把体积预算和 CI 门禁绑定，超阈值必须说明原因并给出回收计划。

### 可背诵话术
“我做构建优化不是为了追求漂亮数字，而是把发布效率、用户体验和成本放在同一张指标表里统一决策。”

## 场景 6：工程效率体系建设（CI/CD、自动化测试、灰度回滚、医疗模板化）

### 1) 业务背景与目标
- 背景：从 0 到 1 承接多家医院定制需求，需求快、变更多，手工回归和手工发布已跟不上。
- 目标：缩短交付周期，降低回归风险，做到“可灰度、可回滚、可审计”。

### 2) 初始方案
- 方案：人工点检 + 手工发版 + 简单冒烟，发布依赖核心同学经验。
- 当时考虑：团队小，先靠人兜底。

### 3) 踩坑表现
- 症状：一次问诊改版后线上出现表单校验回归，周末紧急修复两次。
- 数据：版本交付周期平均 `12 天`，线上回滚率 `18%`，夜间告警频发。
- 用户影响：医院项目经理对版本稳定性信心下降。

### 4) 排查过程
- 过程复盘：回看 3 个月发布记录、缺陷单、回滚原因。
- 关键发现：
  - 缺少分层测试策略，E2E 覆盖低且执行慢。
  - 无灰度能力，发布即全量暴露。
  - 回滚依赖人工操作，步骤易错。

### 5) 根因分析
- 技术根因：
  - 流水线没有质量门禁（单测、E2E、契约测试未绑定发布）。
  - 环境配置漂移，测试环境与生产不一致。
  - 缺少 feature flag，无法按功能粒度开关。
- 流程根因：
  - 发布节奏无“冻结窗口”和风险分级。
  - 复盘仅停留在“谁改坏了”，没有机制改进。

### 6) 解决方案
- 短期止血：
  - 关键链路加 Cypress 冒烟（登录、问诊、处方提交）。
  - 建立一键回滚脚本，缩短故障恢复时间。
  - 高风险需求开启发布冻结和双人复核。
- 长期治理：
  - CI/CD 标准化：`lint + unit(Jest) + contract + e2e(Cypress)` 分层门禁。
  - 灰度发布：按医院、科室、用户组分批放量，可实时回切。
  - Feature Flag 平台化，支持按配置启停功能。
  - 医疗前端模板/低代码化：问诊单、病历页、报告页沉淀成模板，减少重复开发。

### 7) 架构取舍
- 备选方案 A：重投入全量 E2E，追求高覆盖。
- 备选方案 B：测试金字塔 + 风险分级发布（最终选择）。
- 备选方案 C：继续人工回归，强化 checklist。
- 决策理由：
  - A 方案成本高且反馈慢，不适合高频迭代。
  - C 方案无法规模化。
  - B 方案兼顾速度和质量，适配医疗项目节奏。

### 8) 结果量化
- 交付周期从 `12 天` 缩短到 `5 天`。
- 线上回滚率从 `18%` 降到 `4%`。
- 故障恢复时间从 `35 分钟` 降到 `8 分钟`。
- 模板化后新医院首批页面交付效率提升 `40%`。

### 9) 复盘沉淀
- 固化《发布与回滚 SOP》：灰度比例、观测指标、回滚触发阈值。
- 建立质量看板：缺陷逃逸率、自动化覆盖、流水线通过率。
- 把通用能力做成脚手架：权限、审计、日志、表单模板默认内置。

### 10) 面试表达版本
- 30 秒电梯版：
  - “我在医疗项目里把发布从‘靠人’升级到‘靠系统’：分层测试门禁、灰度发布、一键回滚和 feature flag 全部接入流水线。交付周期从 12 天降到 5 天，回滚率从 18% 降到 4%。”
- 3 分钟完整版（STAR）：
  - `S`：多院区并行上线，人工发布和回归带来高风险。
  - `T`：提升交付效率同时降低线上故障。
  - `A`：我先盘点缺陷和发布数据，定义高风险链路；短期上冒烟和回滚脚本，长期搭建 CI/CD 门禁和灰度体系，并推动模板化减少重复开发。
  - `R`：上线节奏更快且更稳，团队从被动救火转向可预测交付。
- 面试官可能追问 3 个问题 + 最佳回答：
  - 问题 1：为什么选择 Cypress 而不是全用 Jest？
  - 回答：Jest 适合逻辑层快速反馈，Cypress 覆盖关键用户流程。两者分工明确，才能兼顾速度和端到端风险。
  - 问题 2：灰度发布如何定义放量节奏？
  - 回答：按风险分级，先内部科室 5%，观察核心指标后再 20%、50%、100%，每阶段都有自动化回滚阈值。
  - 问题 3：模板化会不会限制业务创新？
  - 回答：模板只固化共性能力，业务差异通过 schema 配置和扩展插件实现，既快又不封死扩展性。

### 可背诵话术
“我做工程效率的思路是：先把高风险链路自动化，再把重复劳动模板化，最后用灰度和回滚把线上风险关进笼子。”
